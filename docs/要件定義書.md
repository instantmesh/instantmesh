# 要件定義書：InstantMesh（フェーズ1）

## 1. プロダクト概要
ユーザーが事前設定や面倒な登録なしに、同じ空間（または一時的なグループ）にいるメンバー間で、安全な仮想ローカルネットワーク（Mesh VPN）を即座に構築できるソフトウェア。

* **最大の特徴**: ネットワークを開く人（ホスト）だけがサービスのアカウント（ホストアカウント）を持ち、参加する人（ゲスト）はアカウント登録不要（アカウントレス）で利用できる点。

### 1.1. 解決する課題とターゲット（プライマリ・ペルソナ）
* **プライマリ・ペルソナ**: **ローカルAI／リモート開発者および小規模開発チーム**。ローカルLLM（Dify＋ローカルLLM等）や開発サーバーを、チームメンバーやリモート環境と一時的かつ安全に共有したいが、既存VPNの常設セットアップやポート開放・固定的なアカウント管理を避けたい層。
* **コアな課題**: ローカル開発リソースの共有には「手軽さ」と「セキュリティ」がトレードオフになりがちで、都度のVPN構築・ポート開放・招待管理が負担になる。InstantMeshは「必要なときだけ即席で張り、終われば自動で消える」ネットワークでこれを解消する。
* **注記**: 本ペルソナを唯一の基準とし、プラン・ポート制限・成功指標・訴求はすべてこれに従属させる（ゲーマー／イベント等のカジュアル層は主対象外）。

## 2. スコープと成功指標

### 2.1. やること（フェーズ1の対象）
* デスクトップ3種（Windows / macOS / Linux）向けのアカウントレス即席Mesh VPN。
* ホストのアカウント認証、ルーム作成・待合室承認・キック、エフェメラルなライフサイクル管理。
* 無料／有料のプラン差の**仕様定義**（機能制限の実装を含む。決済は除く）。

### 2.2. やらないこと（フェーズ1のスコープ外）
* **決済・課金の実装**（有料プランの提供機能は仕様として確定するが、課金処理は後続フェーズ）。
* **モバイル（iOS / Android）対応**（VPN審査・OS制約が重いため後続フェーズ）。
* GPUマーケットプレイス等の派生プロダクト（§8 将来展望を参照）。
* マルチAZ／完全冗長化構成（スケール時に移行。§6・§7参照）。

### 2.3. フェーズ1の成功指標
フルファネルKPIは時期尚早とし、**検証仮説1本＋合否基準**に絞る。

* **検証仮説**: 「ローカルAI／リモート開発者は、アカウントレスな即席VPNに対して継続的な利用価値を見出す」。
* **合否基準（仮）**:
    * ターゲット開発者による**ルーム再作成（リピート）率 ≥ 40%**（初回利用者のうち2回目以降を作成した割合）。
    * **P2P（直接）接続成功率 ≥ 80%**（§6 非機能要件）。
    * クリティカルなUXブロッカー（接続失敗・切断の未回復）が主要フローで発生しないこと。

## 3. ユーザー体験（UX）フロー

### 3.1. ホスト（ネットワーク作成者）
1. アプリを起動し、GitHubまたはGoogleアカウントで1クリックログイン（MFAは任意・推奨）。
2. 「ルーム（ネットワーク）を作成」ボタンを押す。
3. 画面に接続用の **QRコード** および **インバイトリンク** が表示される（いずれもホストの公開鍵フィンガープリントを含み、帯域外検証に用いる）。
4. 制限時間（無料版：最大1時間 / 有料版：最大24時間）を設定する。
5. 待合室に届いた参加リクエストを、ゲストの**公開鍵フィンガープリント（SAS）とニックネーム**を確認して個別に承認／拒否する。
6. 運用中に招待URL/QRコードが意図せず漏洩した場合、「招待リンク再発行」ボタンで、既存の接続メンバーを維持したまま新しい招待URLへ即座にローテーションできる（旧URLは即時無効）。不審な既接続ゲストは「キック」で遮断する。

### 3.2. ゲスト（参加者）
1. アプリを起動し、ログインせず「ルームに参加」を選択。
2. ホストから受け取った招待リンク（`instantmesh://join?...`）を入力・貼り付けする（現状ゲスト側のQRスキャン/読み取りは未実装。QRはホスト側の表示のみ）。
3. 一時的なニックネーム（表示名）を入力し、参加リクエストを送信して待合室で待機する。
4. ホストが接続を承認すると、アカウント登録なしで、即座に仮想ネットワークに参加。
5. 割り当てられた専用のプライベートIP（例: `10.0.0.x`）を使い、メンバー間で通信（Ping、割当IP経由のローカル共有、ローカルサーバーへのアクセスなど）を開始。

> 注: 「ファイル共有」は割当IP経由でSMB等のローカル共有に到達できるという接続性の説明であり、専用のファイル共有機能を提供するものではない。

## 4. 機能要件

### 4.1. アカウント・権限管理
* **ホスト**: システムが提供する認証基盤（Amazon Cognitoによるユーザー登録/サインイン）を必須とする。MFA（TOTP）は任意で提供（推奨）。ルームの作成、強制解散、接続承認（待合室フロー）、および不審なゲストのキック（遮断）権限を持つ。
* **ゲスト**: アカウント登録不要。認証・識別はホストが発行した「使い捨てのルームトークン」および一時的な「デバイス識別子（WireGuard公開鍵）」で行う。接続時に一時的なニックネームを入力し、ホストの承認（待合室フロー）を経て接続される。
* **招待トークン強度**: ルームトークンは推測不能とするため **128bit 以上を CSPRNG で生成**する。

### 4.2. ネットワーク（データプレーン）
* **P2P メッシュ通信**: プロトコルは **WireGuard** をベースとし、端末間はエンドツーエンド（E2E）で暗号化する。
* **NATトラバーサル**: カフェのWi-Fiなどの環境でも繋がるよう、STUN技術を用いた「穴あけ（Hole Punching）」を実装する。
* **リレー（フォールバック）**: P2P接続がどうしても失敗した場合のみ、暗号化パケットをそのまま中継するリレーサーバー（DERP 相当の自前リレー。宛先公開鍵ベースの転送で TURN は未使用）を経由させる。
* **鍵交換の完全性（MITM対策）**: QRコード／招待リンクに**ホストの公開鍵（またはそのフィンガープリント）を埋め込み**、シグナリングサーバーを経由しない帯域外（out-of-band）で検証する。これによりシグナリングサーバーが侵害されても中間者攻撃（MITM）を成立させない。ゲストの参加承認時には、ホスト画面に**ゲスト公開鍵の短縮フィンガープリント（SAS）** を表示する。

### 4.3. コントロールプレーン（管理機能）
* **シグナリング**: 接続開始時、端末同士が互いの暗号化公開鍵と一時IPを交換するためのWebSocket等の仲介システム。
* **IPアドレス割当**: ルームごとに `10.0.0.0/24` 等のレンジを確保し、ホストに `.1`、ゲストは参加順に順次割り当てる。キック・退出後のIPは一定時間の猶予後に再利用する。
* **エフェメラル（自動消滅）ライフサイクル**:
    * ホストが「解散」を押す、または設定された制限時間が経過した場合、シグナリングサーバー上のルーム情報を完全に削除する。
    * ルーム消滅時、全クライアントの仮想NIC（ネットワークカード）およびWireGuardの接続設定を自動的に破棄・クリーンアップする。
    * **無通信タイムアウト（Idle Timeout）**: 有料版・無料版問わず、ルーム内で**純アイドル（まったく通信が発生しない状態）が30分継続した場合のみ**、ゾンビセッション排除のため自動的にルームを解散する。通信が継続している限り（長時間の開発共有を含む）解散しない。

### 4.4. 状態遷移・エラー要件
* **待合室フロー（状態遷移）**:
    * 状態は「申請中（待合室）→ 承認 / 拒否 / 無応答タイムアウト失効 / キック済み」。
    * **承認前ネットワーク隔離**: 待合室段階ではWireGuardのピアを追加せず、仮想IPも割り当てない（シグナリング接続のみ）。承認されて初めてネットワークへ参加する。
    * **無応答タイムアウト（仮 120秒）**: ホストが一定時間承認・拒否しない場合、申請を自動失効させ、ゲストに再申請を促す。
    * **同時申請**: 複数ゲストの申請はキューで表示し、ホストが個別に承認／拒否する。
    * **踏み台DoS対策**: 1トークンあたり・1IPあたりの参加申請にレート制限をかける。
* **キック（遮断／再参加ブロック）**:
    * キック時、対象ゲストの**公開鍵とルームトークンをブラックリストに登録**し、WireGuardピアを削除してIPを回収する。
    * トークン失効のみでは新トークンで復帰可能なため、**公開鍵ベースのブロックを主**とし、ルーム存続中は同一鍵の再参加を拒否する。
* **招待URLの即時再発行**:
    * ホストの操作でルーム識別用セッショントークンを即座に再生成し、旧URL/QRコードからのアクセスをすべて無効（404）にする。**承認済みの既接続ピアは維持**される。
    * 注意事項として、**すでに漏洩した「承認済みの相手」の排除には再発行では不十分であり、キックの併用が必要**であることをUI上で明示する。
* **エラー系・例外系**（主要フローで必ずハンドリング・UI提示する）:
    * トークン失効・ルーム消滅時の通知と再取得導線／認証（Cognito）失敗／承認拒否・タイムアウトの通知／P2P確立失敗時のリレーフォールバック／リレー100MB到達（速度制限）の通知／対称NAT等でP2Pもリレーも不能な場合の明示的な接続不可通知／切断からの再接続。

### 4.5. クラウドコスト・悪用対策
* **無料版のポート制限（既定フィルタ／緩和策）**:
  パケット浪費とカジュアルな悪用（SSH/RDP等の常用）を抑えるため、無料版はクライアント側のデフォルトフィルタで特定のWeb・開発用ポートのみを許可する。
  * 許可ポート: `80`, `443`（Web確認）, `3000`, `5000`, `8080`（主要開発サーバー）, および `ICMP`（Ping疎通確認用）。
  * **位置づけ**: 本フィルタはクライアント側実装であり改変によりバイパスされうるため、「物理的な防止」ではなく**緩和策**と明確に位置づける。強制的な悪用抑止はリレー量制限・レート制限・監査ログで担保する。
* **リレー通信量制限**:
  AWSのデータ転送量課金（Data Transfer Out）のスパイクを防ぐため、**リレーサーバー経由の通信は「1接続あたり累計100MB」に到達した時点で64kbpsへ速度制限**する（切断はしない。Ping/軽量API確認など開発用途の軽通信は継続可能）。※P2P直接通信の場合は無制限。
* **レート制限（リソース枯渇対策）**:
  1アカウントあたりの同時ルーム数・ルーム作成頻度に上限を設け、リレー悪用・ルーム量産によるリソース枯渇を防ぐ。

## 6. システム構成・インフラ要件（AWS想定）

| AWSコンポーネント | 役割・選定理由 |
| :--- | :--- |
| **Amazon EC2** (`t4g.small` 等) | **シグナリング／リレーサーバー**。コストパフォーマンスに優れたARMベース（Graviton）を採用。Goで実装したWebSocket通信と、P2P失敗時のリレーパケットを処理。**シグナリングとリレーはコード上論理的に分離**し、スケール時に別インスタンス／マルチAZへ切り出せる設計とする。 |
| **Amazon ElastiCache for Redis** | **ルーム情報の一時管理**。ホストが作成したアクティブなルーム情報と参加ゲストのペアリング情報を保持。RedisのTTL（生存時間）機能を使い、設定時間後の「ルーム自動消滅」を裏側で担保する。 |
| **Amazon Cognito** | **ホストのユーザー認証**。Google / GitHubとのOAuth連携を自前実装せず、セキュアかつ高速に組み込むために採用。MFA（TOTP）を任意で提供。 |
| **Amazon S3（監査ログ）** | 接続メタデータ（監査ログ）を **Object Lock（改ざん防止）＋KMS暗号化＋最小権限アクセス**で保持。通信内容は保持しない（§7 非機能要件）。 |

* **リージョン**: 全コンポーネントを **ap-northeast-1（東京）に固定**し、個人データの越境移転を発生させない。
* **可用性の段階方針**: フェーズ1は最小構成（コスト優先）で開始し、単一障害点は既知リスクとして受容する。スケール時にシグナリング／リレー分離・マルチAZ・冗長化へ移行する。

## 7. 非機能要件

* **開発言語**: サーバー、クライアントともに **Go言語** をベースに構築する。
* **セキュリティ（E2E暗号化）**: サーバー（AWS）側はシグナリングの仲介と暗号化パケットのリレーを行うのみであり、ユーザー間の通信内容（データ）を**復号・閲覧できない**構造にすること。鍵交換の完全性は §4.2 のホスト鍵ピン留め／SAS照合で担保し、シグナリングサーバー侵害時のMITMを防止する。
* **鍵・秘密情報の管理**: クライアントのWireGuard秘密鍵は、メモリロック（mlock）・使用後のゼロ化・スワップ抑止を行い、ディスク／スワップへの露出を防ぐ。
* **監査ログ（法的防衛線）**: 「どのホストアカウントが、いつルームを作成し、どのIPアドレスのゲストが参加したか」という**接続メタデータのみ**を保持する（**通信内容は保持しない**）。
    * 保持期間: **3ヶ月（仮）**。電気通信事業法の該当性判定（後述の並行タスク）を踏まえて確定する。
    * 保護: S3 Object Lockによる改ざん防止、KMSによる暗号化、アクセスは最小権限に限定。
    * **利用規約との整合**: 「記録は技術的に不可能／痕跡ゼロ」といった表現は用いず、「通信内容は復号・閲覧しないが、接続メタデータは保持する」旨をToSに明記する。
* **データ保護・プライバシー**: ゲストのIPアドレス等は個人データとして扱う。ゲストの参加時に**プライバシー通知＋同意**を取得し、保持期間・削除方針を提示する。リージョンは東京固定で越境移転は行わない。
* **可用性・性能（定量目標）**:
    * ネットワーク接続性: 一般的なパブリックWi-Fiにおいて **P2P（直接）接続成功率 80%以上**（必達目標）。
    * P2P接続確立時間: 目標 **10秒以内**（仮）。
    * 想定同時ルーム数・同時接続数の上限を定義し、超過時の挙動（新規作成の制限等）を明示する。
    * 監視・アラート、基本的なDDoS対策（レート制限・接続数制限）を備える。
* **対応プラットフォーム**: フェーズ1は **Windows / macOS / Linux のデスクトップ**を対象とする（モバイルは後続）。

> **並行タスク（法務）**: 他者通信をリレー媒介することによる**電気通信事業法の該当性**（届出要否・通信の秘密・開示対応）は弁護士判定を並行で進め、結果を監査ログ保持期間・開示対応フローに反映する。

## 8. 将来的な展望（C2C GPUプラットフォームへの拡張ビジョン）

本プロジェクト（InstantMesh）は、「アカウントレスかつ手軽に構築できる一時的な仮想プライベートネットワーク（VPN）」として独立したサービスであり続けます。

将来的には、InstantMeshの接続技術（P2P通信、NATトラバーサル、リレー）をSDK / ライブラリとして切り出し、別プロダクトであるC2C GPUシェアリング・マーケットプレイスの通信基盤として内部利用する構想です。

### 8.1. プロダクト構成（2サービス体制）
* **InstantMesh（本プロダクト）**: 即席VPNサービスとして独立して開発・運用・マネタイズする。
* **GPUマーケットプレイス（別プロダクト・別ブランド）**: C2C GPUシェアリングの独立したサービスとして開発する。内部の通信レイヤーにInstantMeshのSDKを利用する。

### 8.2. GPUマーケットプレイスの概要
* **概要**: 余剰GPUを「貸したい人（ホスト）」と「借りたい人（ゲスト）」を直接マッチングするC2Cプラットフォーム。
* **アカウント要件**: InstantMeshとは異なり、GPUマーケットプレイスでは貸し手・借り手の双方にユーザー登録を必須とする。決済（売上の受け取り・利用料の支払い）および不正利用時の本人特定が必要なため。
* **VPN技術の応用**: 借り手が貸し手のGPUマシン（または隔離されたコンテナ）に対して、ポート開放不要でセキュアにアクセスするためのネットワークトンネルとしてInstantMeshのSDKを利用する。
* **追加予定の要素**:
    * **セキュリティ隔離（サンドボックス）**: 貸し手のPC本体や家庭内LANを保護するため、GPUリソースのみをパススルーしたDockerコンテナ等へのアクセス制限。
    * **マッチング＆決済**: GPUスペック、価格、貸出時間に応じたマッチングシステムおよび時間単位の自動決済機能。
    * **検証メカニズム**: 提供されるGPUのスペックや稼働状況（Uptime）を自動検証・評価する機能。

### 8.3. SDK化に向けた設計指針
InstantMeshの開発時から、将来的にコア通信機能をSDKとして切り出せるよう、以下の点を意識して設計する。
* **UIとコアロジックの分離**: VPN接続処理（WireGuard制御、シグナリング、NATトラバーサル）をUIに依存しない独立パッケージ（`pkg/`配下）として実装する。
* **APIの汎用性**: シグナリングやリレーの接続インターフェースを、InstantMeshのUI専用ではなく、外部プロダクトからも呼び出せる汎用的なAPI設計とする。

### A.3. デビルズアドボケイトの反論のうち、検討に値するもの

批判役は「どう直すか」ではなく「そもそも作る/直す価値があるか」を突いた。リーダー評価として是々非々で採否を付す。

#### 採用すべき（鋭く、意思決定に効く）

1. **価値仮説の構造的自壊（異議1・2・4）— 最重要**
   「WTPがある層（開発チーム）は既にアカウントを持ちTailscale等で足りる／刺さるカジュアル層は¥500を払わない」「手軽さはアプリ導入＋管理者権限で相殺されURLを踏むだけのngrok/Cloudflareに劣る」「E2Eの信頼保証とアカウントレスは両立しない」。→ **ペルソナを1つに決めても解けない**という指摘は正当。要件を精緻化する前に価値仮説の検証が要る。

2. **ポート制限は「調整」でなく「削除」を検討（対ビジネス2への反論）**
   E2E暗号化トンネル内をサーバーがポート単位で選別できるなら、それ自体がE2E主張と矛盾。制御はクライアント自己申告依存でバイパス可能。→ 悪用防止としてほぼ無意味かつUX毀損。削除も選択肢。

3. **監査ログとToSは文言修正で直らない二律背反（C4の格上げ）**
   ログを持てば「記録不可能・痕跡ゼロ」訴求が虚偽になり、持たなければ法的防衛線が消える。→ マーケ・法務・技術の三者択一であり経営判断事項。

4. **電気通信事業法を「後で評価」でなく事業成立性の前提リスクへ格上げ（C5）**
   単価¥500のB2Cに届出・通信の秘密・開示対応の負荷は重い。→ 着手前の意思決定材料。

5. **PIN機能自体の要否を問う（対機能5への反論）**
   人間が待合室で承認する設計なら、PINは冗長な第二の壁の可能性。→ 「対策を足す」前に「機能を削る」を先に検討。

6. **見落とされた実現可能性リスク（追加）**
   ①クロスプラットフォーム配布・署名工数（Wintun/EV署名、macOS公証+Network Extension、iOS VPN審査、Android VpnService）、②**決済・課金がスコープに存在しない**のに有料版を謳う、③スケジュール・体制・コストの前提が皆無。

#### 割り引いて扱うべき（一理あるが行き過ぎ）

- **「KPI完備要求は過剰」（対ビジネス4）**: フルファネルKPIは時期尚早に同意。ただし「フェーズ1の成功をどう判定するか」の定義は不可欠。→ フルKPIではなく“検証仮説1本＋合否基準”に絞る折衷。
- **「IP割当は軽微」（対機能10）**: 実装詳細として軽微化は妥当。ただしゲスト数上限・キック後IP再利用と絡むため、データ設計の一項目としては残す。
- **「GPUビジョンは踏み台になっていない（異議5）」**: 技術的連続性の弱さは正当な警告。フェーズ1の投資判断を「独立事業の採算」で行うべきという主張として扱えば十分。

#### リーダー留保

批判役の全体トーンは「作るな」寄りだが、最終判断はユーザーのもの。批判役の価値は「作らない理由」ではなく「作るなら何を先に確かめるべきか」を明確にした点にある。

### A.5. 各担当レビュー詳細

#### A.5.1 ビジネス・ユーザー視点

**重大**
- ターゲット／ペルソナがドキュメント間で矛盾（要件§1／vision§2／競合分析§1.1,1.2,3.2）。要件定義書自体に課題・ターゲット記述なし。

#### A.5.2 機能要件

**重大**: ①最大ゲスト数の要件が全面的に欠落（3章/4章表 L60-73）。②プラン構成・制限値がドキュメント間で不整合。③キック機能の仕様がほぼ未定義（L33のみ、再参加ブロック不明）。④待合室フローに拒否/タイムアウト/同時申請が欠落。⑤PINにブルートフォース対策と検証タイミング定義なし（シーケンス図にPIN照合なし）。

**重大**
- **R-1** シグナリングサーバー侵害時の鍵交換MITM（鍵配布経路がシグナリング自身、フィンガープリント検証なし、「侵害されても盗聴不可能」の主張が崩れる／アーキ4.1 L157-159）。

**軽微**: D-1 数値不整合。D-2 ゲストToS非同意。D-3 100MB到達後の挙動。D-4 再発行しても承認済み漏洩相手は排除不可。

#### A.5.4 デビルズアドボケイト（批判役）

**前提への異議**: ①アカウントレスは金を払う層に無価値、刺さる層は金を払わない（WTPと差別化の分離）。②「QRを読むだけ」は虚偽（アプリ導入＋管理者権限が必須、ngrok/Cloudflareに手軽さで劣る）。③コア用途とインフラ制約が衝突（大容量共有が100MBで即死、コアターゲットほど対称NATでP2P失敗）。④セキュリティを厳密化するほどアカウントレスが死ぬジレンマ（R-1は修正不能）。⑤GPUビジョンは技術的に繋がっていない（GPU用途は両者アカウント必須でフェーズ1固有機能を使わない）。

**代替案（D案）**: コア技術とクロスプラットフォームアプリの自作前に、既製品（Tailscale tsnet / NetBird / Cloudflare Tunnel）の薄いラッパーで価値仮説（WTP）を10〜20人に検証。最初の投下先は「技術が動くか」でなく「誰が金を払うか」。

### A.6. 総括

技術・アーキテクチャの筋は通っているが、視点を変えるたびに同じ穴が浮かんだ。特に「プラン不整合＋ゲスト数欠落」「待合室/キック等の状態遷移欠落」「PIN耐性」「監査ログ⇔ToS矛盾」「ターゲット矛盾」は複数レビュアーが独立検出した構造的欠陥。

また、要件の穴の多く（PIN・ポート制限・待合室・ログ矛盾）は「その機能を作らなければ発生しない」もので、精緻化するほど自己矛盾が深まりかねない。

**推奨する順序**: ゲート0（特に G0-1 ペルソナ確定・G0-2 WTP検証・G0-3 法務判定）を先に通し、その結果を踏まえてフェーズ1a/1bの要件修正に入る。

### B.4. ドラフト確定（既定値・要件へ反映）

技術的最適解が定まっているため、以下の内容で要件へ反映した。

#### B.4.1 待合室フローの状態遷移（C7 / A-3）
- 状態: 申請中（待合室）→ 承認 / 拒否 / 無応答タイムアウト失効 / キック済み
- **無応答タイムアウト = 仮120秒** で申請を自動失効。ゲストに再申請を促す
- **同時申請**: 複数ゲストの申請をキューで表示し、ホストが個別に承認/拒否
- **レート制限**: 1トークンあたり・1IPあたりの参加申請にレート制限（踏み台DoS対策）
- **承認前ネットワーク隔離**: 待合室段階ではWireGuardピア未追加・仮想IP未割当（シグナリング接続のみ）

#### B.4.2 キック（再参加ブロック）（C8）
- キック時に**ゲストの公開鍵＋ルームトークンをブラックリスト登録** → ピア削除・IP回収
- トークン失効のみでは新トークンで復帰可能なため、**公開鍵ベースのブロックを主**とする（ルーム存続中は同一鍵の再参加を拒否）

#### B.4.3 なりすまし補強（🟡 ★★★）
- 承認画面にゲスト公開鍵の**短縮フィンガープリント(SAS)** を表示。ニックネームは自己申告のため「未検証」ラベルを付す
- ニックネームは長さ・使用文字をバリデーションし、ルーム内重複時は自動サフィックス（例 `#2`）を付与

#### B.4.4 その他（🟡/🟢 まとめ処理）
- **エラー系要件節を新設**: トークン失効UI／認証失敗／再接続／ポート弾き通知／100MB到達通知
- **招待URL再発行の状態遷移**: 旧トークンを即時失効(404)し承認済みピアは維持。ただし**既に漏洩した承認済み相手の排除はキック併用が必要**と明記（D-4）
- **IP割当ロジック**: `10.0.0.0/24` 等からホストに `.1`、ゲストは順次割当。キック後のIPは一定時間後に再利用
- **Idleタイムアウト**: 「純アイドル（無通信）30分」のみを対象とし、通信中（長時間の開発共有含む）は解散しないことを明記
- **レート制限（ルーム量産）**: 1アカウントあたりの同時ルーム数・作成頻度に制限（リレー悪用・リソース枯渇対策）
- **秘密鍵管理**: メモリロック(mlock)／使用後ゼロ化／スワップ抑止をNFRに明記
- **「やること/やらないこと」節を新設**
- **「例:」表記の確定値化**: Idle=30分（確定）、リレー速度制限=64kbps（確定）、監査ログ保持=3ヶ月（仮確定）、P2P成功率80%=必達目標として記述
- **ファイル共有の位置づけ**: 「割当IP経由でローカル共有(SMB等)が可能」という接続性の説明にとどめ、機能保証ではないことを明確化
- PIN削除に伴い「パスコード（オプション/標準）」関連記述を削除

### B.5. 批判役論点の採否（リーダー判断）

| 論点 | 採否 | 判断 |
|------|:---:|------|
| 価値仮説の構造的自壊（WTPと差別化の分離） | △ | 指摘は正当と認めつつ、事前検証は行わず開発に進む（G0-2）。市場リスクは受容 |
| ポート制限は「削除」を検討 | △ | 削除ではなく「緩和策として正確化」を選択（B-3） |
| 監査ログとToSは文言修正で直らない | ✅ | ログ保持を選び、ToS文言を実態に合わせて修正（G0-4） |
| 電気通信事業法を前提リスクへ格上げ | ✅ | 並行タスク化して評価（G0-3） |
| PIN機能自体の要否を問う | ✅ | 削除を採用（C6） |
