# プロダクトビジョン＆ユースケース戦略：InstantMesh

## 1. 解決するコアな課題（Why InstantMesh?）
現在のリモート開発、特に「ローカルAI（Dify + ローカルLLM）開発」の現場において、重大なセキュリティのジレンマが存在している。

* **ユーザーの背景**:
  データを外部クラウド（OpenAI等）に送信せず、機密情報を100%手元に閉じ込めるためにローカルLLMやDifyを導入している。
* **現在の課題（ngrokの罠）**:
  社外のテスターやスマホ実機から一時的に検証する際、公式に推奨されている「ngrok」等を利用せざるを得ない。しかし、これによって「全世界にパブリックURLが露出する」「中央集権の海外サーバーを経由する」というリスクが発生し、ローカルに閉じ込めていたはずの機密レベルが崩壊している。

## 2. ターゲットセグメント（フェーズ1のペルソナ）
* ローカル環境でDify、Flowise、または自作のローカルLLMアプリを検証しているエンジニア。
* クライアントのデータを扱うため、 ngrok などのパブリックURL発行ツールを情シス（情報システム部門）から禁止されている開発チーム。
* ハッカソンやコワーキングスペースなどのパブリックWi-Fi（クライアント間通信禁止環境）で、隣のメンバーに安全かつ爆速でローカルサーバーを共有したい開発者。

## 3. InstantMeshが提供する圧倒的価値
InstantMeshは、このセキュリティのジレンマを**「アカウントレス」「使い捨て（エフェメラル）」「完全P2Pクローズド接続」**というアプローチで解決する。

* **全世界への隠蔽（URLレス）**:
  公開URLは一切発行されない。ホストの生成したQRコード/リンクから、ホストが承認した特定のゲストだけが同じ仮想LAN（ルームごとに 10.x 系プライベートレンジから /24 を払い出す）に入り、Dify等（ポート80/8080等）にアクセスできる。
* **データ主権の死守（完全P2P）**:
  シグナリング完了後は端末間で直接暗号化パケット（WireGuard）をやり取りするため、データが外部サーバーを通過せず、機密情報が100%担保される。
* **無料版のポート制限による安全担保**:
  無料版の通信許可ポートをWeb/主要開発ポート（80, 443, 8080, 3000等）に限定することで、万が一の悪用（SSH/RDP等によるハッキングバックドア化）を標準で遮断する。これはクライアント側の既定フィルタによる緩和策であり（改変クライアントでは回避されうる）、E2E 暗号化の P2P 直通トラフィックをサーバー側で強制遮断するものではない。
